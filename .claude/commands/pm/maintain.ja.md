# Skill: /pm:maintain
**ロール**: PM / Dev / BE
**目的**: 保守フェーズ向けの最小ワークフロー — active スプリントの overhead なしに十分な統制を提供。

---

## コンテキスト

保守フェーズはアクティブ開発と異なる:
- スプリントプランニング / バックログ整理は定期的でない
- 依頼はアドホック — JP 顧客はいつでもバグレポートをメール
- チームが小規模 — 通常 1-2 開発者 + 1 BE パートタイム
- JP 受託: SLA / 保守契約が応答時間と修正時間を規定
- billing / 月次報告のため履歴追跡が重要

---

## 保守ワークフロー

```
JP バグレポート / 依頼
    → トリアージ（分類 + 優先度）
    → 修正（tiny: 直接パッチ / normal: dev:analyze → implement）
    → QA 検証
    → リリース（ホットフィックスまたは月次バッチ）
    → 月次保守報告書（JP へ送付）
```

---

## 実施手順

### ステップ 1 — ゲート: 依頼種別の特定

`AskUserQuestion` tool で確認:

- **種別**: バグ修正 / 軽微改善 / パフォーマンス問題 / セキュリティパッチ / JP からの問い合わせ
- **ソース**: JP 顧客報告 / ユーザー報告 / 監視アラート / 内部発見
- **SLA**: SLA 義務あり? 要求応答時間?

**確認待ち。**

### ステップ 2 — トリアージ

保守優先度マトリクスで分類:

```
## トリアージ: [問題名]

**優先度**: P1 / P2 / P3 / P4

| 優先度 | 定義 | 目標修正時間 |
|----------|-----------|----------------|
| P1 — Critical | 本番停止 / データ損失 / セキュリティ侵害 | 即時修正、当日中 |
| P2 — High | 機能不全、回避策困難 | 1-3 日以内 |
| P3 — Medium | 機能不全、回避策あり | 1-2 週以内 |
| P4 — Low | UI 不具合 / 軽微な不便 | 月次リリースにバッチ |

**分類**: [P1/P2/P3/P4]
**理由**: [...]
**担当**: [Dev name]
**修正期限 (JST)**: [Date]
```

### ステップ 3 — 優先度別修正ワークフロー

#### P1 — Critical（ホットフィックス）

```
P1 ホットフィックスプロトコル:
1. BE に即時連絡 → BE が 30 分以内に JP 顧客に通知
2. Dev 修正（tiny レーン — 直接パッチ使用）
3. QA クイック検証（スモークテスト）
4. ホットフィックスを即時デプロイ
5. ポストモーテム: docs/reports/incident-[date].md に記録
6. 障害報告書を JP 顧客に送付（/ops:incident 使用）
```

#### P2/P3 — Normal 修正

簡略化された開発ワークフロー適用:

```
analysis.md 必要なタスク（normal risk）:
  /dev:analyze → /dev:implement → /dev:review → /dev:pr → merge → deploy

小規模パッチ（tiny）:
  直接修正 → /dev:review → /dev:pr → merge → 月次リリースにバッチ
```

#### P4 — 月次リリースにバッチ

`docs/maintenance/[YEAR-MONTH]-backlog.md` に記録:

```markdown
# 保守バックログ: [YYYY-MM]

| ID | 内容 | 報告者 | 優先度 | ステータス |
|----|-------|----------|----------|--------|
| M-001 | [...] | [JP client / internal] | P4 | Open |
```


```
|---|---------|---------|
| 1 | このタスクを月次報告書に記載? | A: あり — バックログに追加済 / B: なし — P1/P2 ホットフィックス、別途報告 / C: その他: ___ |
```

---

## ルール

- P1: 修正完了前に JP 顧客に通知 — 沈黙しない
- 全修正にはチケット（GitHub Issue）必要 — どんなに小さくても — billing と月次報告のため
- 月次報告書は翌月最初の営業日（JST）に送付
- 月次工数が保守契約 limit を超えた場合 → PM に即時アラート、CR 交渉
