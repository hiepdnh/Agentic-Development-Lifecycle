## コンテキスト

ブリッジエンジニアは日本の顧客と VN 開発チームの橋渡し:
- **入力**: JP からの要件（日本語または英語） — メール、議事録、原本仕様書
- **出力**: VN 開発者向け仕様（ベトナム語）+ JP 顧客向け成果物（日本語）
- **基準**: 日本 SI 基準文書 — 設計書、単体テスト仕様書、成果物

## 実施手順

### ステップ 0 — 用語集ロード（必須）

用語を分析/翻訳する前、`templates/jp-vn-en-glossary.md` を読み JP↔VN↔EN 標準用語集をロード。

スキル全体（ステップ 1→4）で技術/業務用語に遭遇した場合:
- 用語集の対応語を優先使用
- 用語集に **ない** 場合 → `[GLOSSARY?]` マーク、ステップ 4 で追加提案

### ステップ 1 — ゲート: 作業種別の特定

```
## JP 顧客からの要件を受領。

作成必要な文書:
- [ ] 基本設計書 — 高レベル仕様、開発前に JP 確認
- [ ] 詳細設計書 — VN 開発者向け詳細仕様
- [ ] 単体テスト仕様書 — JP 標準フォーマットのテストケース
- [ ] 変更依頼 — 変更要求 + 影響分析 → **`/be:changerequest` を代用** して完全な承認証跡確保
- [ ] 障害報告書 — バグ/インシデント報告

入力言語: JP / EN / VN
出力必要: バイリンガル JP+VN / JP のみ / VN のみ

続行確認?
```

**確認待ち。**

#### 3d. バイリンガル HTML 成果物（JP 顧客向け）

`templates/html-bilingual.html` から `docs/tasks/[TASK-ID]/deliverable.html` 生成 — 日本顧客への転送時のメインフォーマット:

- ヘッダ: title JP/VN、TASK-ID、JST タイムスタンプ、ラベル「成果物」
- 各コンテンツ項目に対し `<div class="row">` 2 カラム（JP 左、VN 右）作成
- 用語集にある JP 技術用語 → `<span class="glossary" data-tooltip="JP=VN">` でラップ、ホバーで翻訳表示
- 各行にコピーボタン（`data-copy-parent`）
- `@media print`: A4 印刷美麗、2 カラム維持、コピーボタン非表示 — Cmd+P → PDF 保存で正式版作成可

HTML はコミットしない（`.gitignore` に `docs/tasks/**/*.html` 設定済）。正式版送付時 → PDF エクスポート後メール/Issue 添付。

```
✓ 生成完了:
  - docs/tasks/[TASK-ID]/requirements.md   (VN 開発者向け、commit)
  - docs/tasks/[TASK-ID]/design-jp.md       (JP 顧客にメール送付、commit)
  - docs/tasks/[TASK-ID]/deliverable.html   (送付前の社内レビュー用、commit しない)
```

## ルール

- JP 要件が不明な時、自己解釈せず即座に質問
- JP 技術用語（ログイン、トークン...）→ 原語維持または用語集の標準語使用
- VN 仕様は VN 開発者が JP 原本を読まずに作業可能なレベル
- JP 文書は JP 顧客が VN 仕様を読まずに確認可能なレベル
