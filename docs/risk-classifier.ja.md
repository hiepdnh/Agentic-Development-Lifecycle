# リスク分類器（Risk Classifier）

集中型のリスク分類ゲート。`/dev:analyze` と `/dev:implement` の冒頭で、すべてのタスク開始前に実行する。

---

## 入力タイプ（Input Types）

リスク分類の前に、入力タイプを判定する:

| タイプ | 説明 | 例 |
|------|-------|-----|
| **new-spec** | まだストーリーが存在しない新規仕様 | 顧客からの新規 JP 要件 |
| **spec-slice** | 既存の大型仕様の一部 | 進行中 Epic の子ストーリー |
| **change-request** | 既存挙動の変更 | バリデーションロジックの変更依頼 |
| **maintenance** | バグ修正、依存関係更新 | QA または本番からのバグレポート |
| **new-initiative** | 能動的な技術改善 | リファクタリング、パフォーマンス改善 |
| **framework-improvement** | このフレームワーク自体の改善 | 新スキル追加、テンプレート更新 |

---

## リスクチェックリスト（10 項目）

タスクに該当する項目をマーク:

| # | 質問 | YES → リスク上昇 |
|---|---------|---------------------|
| R-01 | 認証または認可を変更するか? | +high |
| R-02 | 既存レコードに影響するデータモデル変更か? | +high |
| R-03 | 新規外部プロバイダ（API、決済、OAuth）統合か? | +high |
| R-04 | バリデーションを弱体化または安全装置を除去するか? | +high |
| R-05 | パブリック API を変更（破壊的変更）するか? | +high |
| R-06 | 共有インフラ（config、CORS、env）に影響するか? | +normal |
| R-07 | 新規機微データ（PII、決済、健康情報）を保存するか? | +high |
| R-08 | 複数画面 / 複数ユーザーロールの挙動を変更するか? | +normal |
| R-09 | 影響範囲の既存テストカバレッジがないか? | +normal |
| R-10 | クロスドメイン影響（他チーム所有モジュールに影響）があるか? | +normal |

---

## レーン割り当て（Lane Assignment）

### Tiny レーン
**条件**: チェックリスト全項目が NO、変更が小規模かつ自己完結。

**手順**:
- 直接パッチ、ストーリー文書不要
- `analysis.md` 不要
- 最小限の検証: ハッピーパス自己テストのみ

**例**: タイポ修正、ボタン色変更、コピーテキスト更新、バリデーションメッセージ修正。

### Normal レーン
**条件**: R-06/R-08/R-09/R-10 のいずれかが YES、ただし R-01〜R-05 はすべて NO。

**手順**:
- `docs/tasks/[TASK-ID]/analysis.md` を作成
- フルワークフロー実施: `/dev:analyze` → `/dev:implement` → `/dev:pr`
- 検証: セルフテスト手順付き `verification.md` 作成

**例**: フォームへの新フィールド追加、新 API エンドポイント追加、モジュールリファクタリング。

### High-Risk レーン
**条件**: R-01〜R-05 のいずれかが YES、または下記 Hard Triggers に該当。

**手順**:
- **即停止** — シニア / テックリードの確認を得るまで実装しない
- 明確なリスクセクション付き `analysis.md` を作成
- PR 作成前に `/sec:review` 必須
- 大規模な設計変更を含む場合は `/arch:review` 必須
- JP 成果物の場合、顧客送付前にブリッジエンジニアレビュー

---

## ハード High-Risk トリガー（Hard Triggers）

以下の変更は **チェックリスト不要で自動的に high-risk** に分類される:

- 認証 / 認可ロジックの変更
- データ損失の可能性があるデータモデル変更（カラム削除、型変更）
- 外部決済プロバイダの統合
- 暗号実装の変更
- 新規 PII / 健康データの保存
- 既存バリデーション / 安全装置の削除またはバイパス

Hard Trigger 検出時 → `assets/ask-first-gates.md`（または `.ja.md`）のフォーマットで `⚠️ Ask First Gate` を表示。

---

## スキル内での使用方法

```
## リスク分類 — [TASK-ID]

**入力タイプ**: [new-spec / spec-slice / change-request / maintenance / ...]
**リスクチェックリスト**: R-01 ✅ R-02 ❌ R-03 ❌ ...（YES の項目のみ列挙）
**レーン**: tiny | normal | high-risk
**理由**: [1 文での説明]
```

このブロックを各 `analysis.md` の冒頭、および `/dev:implement` の最初のゲートに貼り付ける。
